VIEW -> API -> DB EXECUTION RULES
============================================================

Rule A: For every UI screen load
- Call business endpoint first (data model).
- Then call presentation endpoint for same Blade path.
- Merge as:
  - content/data -> from business endpoint
  - ui contract/tokens/components -> from presentation endpoint

Rule B: Presentation endpoint template
- /api/v2/mobile/presentation/views/{segment}/{path-without-.blade.php}
Examples:
- site faq view:
  /api/v2/mobile/presentation/views/site/faq
- portal support thread view:
  /api/v2/mobile/presentation/views/portal/support/thread
- shop product view:
  /api/v2/mobile/presentation/views/shop/product
- v2 rentals view:
  /api/v2/mobile/presentation/views/v2/rentals

Rule C: Primary business endpoints by module
- Site bikes: /api/v2/mobile/bikes*
- Site rentals: /api/v2/mobile/rentals*
- Site shop: /api/v2/mobile/shop/products*
- Site spareparts: /api/v2/mobile/spare-parts/*
- Site forms: /api/v2/mobile/mot/*, /finance/*, /contact/*
- Club: /api/v2/mobile/club/*
- Portal: /api/v2/mobile/portal/*
- Unified enquiries: /api/v2/mobile/enquiries*
- Chat:
  - customer /api/v2/mobile/customer/support/*
  - staff /api/v2/mobile/staff/support/*

Rule D: Direct DB requirement
- Dynamic modules are DB-backed via models/services.
- Static/editorial-only pages remain static payloads but still exposed via API for mobile rendering consistency.
