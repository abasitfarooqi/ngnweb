 Flux Admin Panel — Implementation Plan

 Context

 The existing Backpack admin panel (98 CRUD controllers at /ngn-admin) is operationally fragmented — staff must navigate many disconnected
 pages to complete a single business process. The goal is to build a new, parallel admin panel called flux-admin using Livewire 4 + Flux
 Pro + Tailwind that unifies these fragmented CRUDs into coherent operational modules. The existing Backpack panel is read-only from a
 migration perspective — no files in app/Http/Controllers/Admin/ or routes/backpack/ will be touched.

 The new panel uses the same database, same models, same auth system (Spatie Permission, admin middleware, check.admin.access middleware,
 User model with is_admin flag). It is a separate UI layer on top of the existing data.

 Phase 2: Reusable Component Library

 All custom components at app/Livewire/FluxAdmin/Components/ with views at resources/views/livewire/flux-admin/components/.

 2.1 DataTable Component

 - Wraps <flux:table> with server-side search, sort, pagination
 - Properties: $search (debounced 300ms), $sortField, $sortDirection, $perPage, $filters
 - Uses WithPagination trait
 - Supports column definitions via a columns() method in child classes
 - Search across multiple fields via LIKE with $searchableFields array
 - Renders <flux:table.column> headers with sortable, <flux:table.row> for each record

 2.2 FilterBar Component

 - Horizontal bar of filter controls (selects, date pickers, toggles, text search)
 - Accepts filter definitions array; emits filter-change events to parent DataTable
 - Uses <flux:select>, <flux:date-picker> (Pro), <flux:switch>

 2.3 EntityPicker Component

 - Wraps <flux:autocomplete> (Flux Pro) for selecting Customer, Motorbike, ClubMember, etc.
 - Configurable: model class, search fields, display format, value field
 - Server-side search with debounce
 - Replaces all Select2-based pickers from the existing JS files (get-customer-details-inline.js, get-motorbike-details-inline.js)

 2.4 InlineEditor Component

 - Toggle between display/edit mode on click
 - Save on blur/Enter, cancel on Escape
 - Validates on save, emits inline-saved event
 - Replaces inline_stock_edit.js and sp_inline_stock_edit.js

 2.5 Blade Components (non-Livewire)

 - resources/views/components/flux-admin/summary-header.blade.php — entity header with title, subtitle, status badges, stat cards, action
 buttons
 - resources/views/components/flux-admin/stat-card.blade.php — label + value + optional trend
 - resources/views/components/flux-admin/slide-over.blade.php — wraps <flux:modal variant="flyout" position="right">

 2.6 GlobalSearch Component

 - Wraps <flux:command> (Flux Pro) — triggered by Cmd+K
 - Searches across: motorbikes (reg_no, VIN), customers (name, email, phone), finance apps (ID), rental bookings (ID), PCN cases
 (pcn_number), club members (name, phone)
 - Uses debounced server-side queries

 ---
 Key Architectural Decisions

 1. Full-page Livewire components for all routes (same pattern as existing Portal/Dashboard.php)
 2. Nested #[Lazy] Livewire components for tabs — each tab manages its own data independently
 3. No JavaScript framework — all interactivity via Livewire + Alpine.js (ships with Livewire)
 4. Same models, no new tables — CrudTrait on models is harmless outside Backpack context
 5. Same auth, same permissions — reuse the existing 13 roles and 87 permissions via @can() and auth()->user()->hasRole()
 6. Module-based navigation — 7 main modules instead of 98 separate CRUD pages

 ---
 Verification Strategy

 For each module after implementation:
 1. Data accuracy — compare record counts and field values with the existing Backpack CRUD list views
 2. Search — verify search finds the same records as Backpack's search
 3. Cross-module links — click through motorbike → rental → customer → PCN chains
 4. Business logic — for Finance, verify contract type mutual exclusion works; for Club, verify balance calculations match
 5. Permissions — log in as different roles and verify visibility gates work
 6. Performance — check page load times and query counts via Laravel Debugbar
